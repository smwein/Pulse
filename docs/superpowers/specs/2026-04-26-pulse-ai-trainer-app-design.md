# Pulse — AI Personal Trainer iOS App

**Status:** Draft, ready for implementation planning
**Date:** 2026-04-26
**Audience:** Solo developer building this app for personal use, designed to allow opening up to other users later without major rewrites

---

## 1. Product Summary

Pulse is a native iPhone app that delivers an AI-coached personal training experience inspired by Future.co — but with a single AI coach (not a human trainer) who plans, adapts, and accompanies every session. Each workout is generated by an LLM based on the user's profile, recent training history, and physiological signals from Apple Health. After every session, the user submits structured feedback that the AI consumes to adapt the next session in real time.

The app's defining UX moment is the **post-workout feedback → AI adaptation loop**: rate the session, see what the AI changed for tomorrow, watch the plan evolve over time.

The full UI design (screens, colors, typography, copy, interaction behaviors, coach personalities) is already specified in the `design_handoff_pulse_workout_app/` folder at the project root and in its `README.md`. This document does **not** duplicate that. It covers everything the design handoff doesn't: system architecture, iOS code structure, AI integration, content pipeline, persistence, networking, and cross-cutting concerns.

### Scope of v1

- **Vertical slice:** 5-step onboarding → coach selection → home/today's workout → AI plan generation (today only initially, growing to weekly) → in-workout (data-dense screen) → 3-step Complete feedback flow → next-session adaptation
- **Out of scope for v1:** Calendar grid, full Exercise Library browser, Stats/Progress dashboard, full Coach Chat. These screens exist in the design handoff but are deferred to v1.1.
- **Single user.** No auth, no multi-tenant. Designed so that adding multi-user later requires backfilling one optional `userID` field, not a rewrite.

| **Observability** | Sentry (free tier) for iOS crashes; Cloudflare Workers Analytics for proxy traffic; in-app debug panel reachable via 5-tap gesture on profile avatar. |

| **Multi-user readiness** | Every SwiftData model carries an optional `userID: UUID?` field, nil for now. Schema unchanged when multi-user is added. |

## 3. System Architecture

```
┌──────────────────────────────────┐         ┌──────────────────────────────┐
│  iPhone (SwiftUI)                │         │  Apple Watch (SwiftUI)       │
│  ─────────────────               │  WCSession  │  ───────────────         │
│  • UI screens                    │ ◄──────► │  • Standalone workout app   │
│  • SwiftData (source of truth)   │  HR / sets │  • HKWorkoutSession         │
│  • Repository layer              │         │  • Streams HR ~1 Hz          │
│  • HealthKit reader/writer       │         │  • Logs sets via taps         │
│  • Builds prompts, calls proxy   │         │                              │
│  • Plays exercise demos (AVPlayer) │       │                              │
└──────┬───────────────────────────┘         └──────────────────────────────┘
       │ HTTPS (POST + SSE-streamed)
       ▼
┌──────────────────────────────────────────────────────────┐
│  Pulse Proxy (1 Cloudflare Worker, ~80 lines TS)         │
│  • Holds ANTHROPIC_API_KEY in Worker secret              │
│  • Validates X-Device-Token shared secret                │
│  • Forwards to api.anthropic.com, streams SSE back       │
│  • Logs request shape, model, tokens, latency            │
│  • No business logic, no database, no state              │
└──────┬─────────────────────────────────────────┬──────────┘
       │                                         │
       ▼                                         ▼
   Anthropic Messages API                  (separately)
   Sonnet 4.6 + Opus 4.7              Cloudflare R2 (public bucket)
                                       • Exercise demo MP4s + posters
                                       • System-prompt JSON (hot-loaded)
                                       • Exercise catalog manifest
                                       Fetched directly by iOS over HTTPS
```

### Data flows

**Plan generation (today's workout):**
1. iOS gathers context: profile, last 5 sessions (from SwiftData), 7-day HealthKit summary, current plan
2. iOS constructs an Anthropic Messages API request body (system prompt + user message) and POSTs it to the Worker
3. Worker validates device token, forwards to Anthropic with `stream: true`, pipes the SSE response back
4. iOS parses the stream: `⟦CHECKPOINT: ...⟧` markers render as mono console lines (matches the prototype's "thinking state" UX); the final ` ```json ... ``` ` block is parsed and validated against the `WorkoutPlan` Codable schema
5. Plan persisted to SwiftData; UI transitions to result card

**In-workout:**
1. User taps "Start" on iPhone → today's workout payload pushed to Watch via `WCSession.transferUserInfo`
2. Watch starts `HKWorkoutSession` (`activityType` derived from workout type)
3. Watch streams HR samples to iPhone at ~1 Hz via `WCSession.sendMessage`; iPhone updates the live HR card / zone display
4. User taps "Log set" on Watch (or iPhone) → set log entry sent via `transferUserInfo` (queued, reliable)
5. iPhone writes set log to SwiftData, advances `SessionStore`, transitions to rest phase
6. On last set of last exercise → end `HKWorkoutSession` (writes one `HKWorkout` to HealthKit) → navigate to Complete flow

**Post-workout adaptation:**
1. User completes 3-step feedback flow → `WorkoutFeedback` object built
2. iOS persists feedback to SwiftData (no network needed for the user to feel "done")
3. iOS calls Worker with adaptation prompt (current plan + feedback + recent context)
4. Stream renders thinking-state animation, then arrives at result phase
5. iOS receives structured plan diff + adjustment list; persists `Adaptation` record; UI shows "Here's what changes" with the *real* adjustments (no client-side derivation)

**Coach chat (deferred to v1.1; designed for future):**
- Streaming Sonnet 4.6 with the active coach's system prompt
- History trimmed to last ~30 messages with rolling summary; prompt-cache breakpoint after every coach reply

### What lives where

| Component | Responsibilities | Notes |
|---|---|---|
| **iPhone app** | UI, business logic, prompt assembly, SwiftData persistence, HealthKit read/write, video playback, Watch coordination | Source of truth for all user data in v1 |
| **Watch app** | Workout session lifecycle (`HKWorkoutSession`), live HR streaming, set logging UI | Three screens only: Idle, Active Set, Rest |
| **Cloudflare Worker** | Authenticate via shared secret, forward Anthropic API calls, stream SSE back | Stateless, no DB, no business logic |
| **Cloudflare R2** | Hosts: exercise MP4s, exercise poster JPEGs, exercise catalog manifest JSON, system prompt JSON | Public bucket; iOS fetches directly |
| **Anthropic API** | LLM inference for plan generation, adaptation, (future) chat | Sonnet 4.6 + Opus 4.7 |

## 4. iOS App Structure

### Module layout

Single Xcode project with local Swift Packages for module boundaries. Each package compiles and tests independently.

```
PulseApp/                          ← SwiftUI app target (entry point only)
Packages/
├── DesignSystem/                  ← Tokens (oklch colors, fonts, radii, motion),
│                                    primitives (Card, Pill, Button, CoachAvatar,
│                                    Ring, ExercisePlaceholder)
├── CoreModels/                    ← Pure value types: Workout, Exercise, Coach,
│                                    Plan, WorkoutFeedback, SetLog. No deps.
├── Persistence/                   ← SwiftData stack, model containers, migrations.
│                                    Wraps @Model entities.
├── Networking/                    ← APIClient (URLSession async/await + SSE),
│                                    auth interceptor, typed endpoints,
│                                    SSEStreamParser
├── HealthKitClient/               ← Wraps HKHealthStore. Read activity/HRV/sleep
│                                    summaries. Write completed workouts.
├── WatchBridge/                   ← WCSession singleton + typed message
│                                    envelopes (phone-side counterpart)
├── Repositories/                  ← Plan, Workout, Feedback, Coach, Chat
│                                    repositories. Compose Networking + Persistence
│                                    + HealthKitClient. UI only talks to these.
├── Features/
│   ├── Onboarding/                ← 5-step quiz + coach selection
│   ├── Home/                      ← Today's workout hero, week strip, weekly ring
│   ├── PlanGeneration/            ← Inputs + thinking-state stream + result card
│   ├── WorkoutDetail/             ← Hero + stats + blocks list + Start CTA
│   ├── InWorkout/                 ← Data-dense session screen.
│   │                                Owns SessionStore (@Observable).
│   ├── Complete/                  ← 3-step feedback + adaptation flow
│   └── CoachChat/                 ← Deferred to v1.1
└── AppShell/                      ← Tab bar, navigation, root environment,
                                     theme provider (dynamic --accent-h),
                                     debug panel
```

### Patterns

- **State management:** SwiftUI + `@Observable` macro (Swift 5.9+). One `@Observable` store per feature, injected via `@Environment`. The `SessionStore` for the in-workout screen owns: `idx` (current exercise index), `setNum` (current set 1..N), `phase` (`.work` | `.rest`), `secs` (timer for current phase), `playing` (bool), `setLog` (`[ExerciseID: [SetNum: SetEntry]]`), `hr` (current heart rate). This matches the state model called out in the design handoff README.
- **Persistence:** SwiftData for all client-side state. Plans, workouts, set logs, feedback drafts, message history. Treated as a cache of server truth in spirit, but in v1 the app *is* the truth (no server DB).
- **Repository pattern:** UI never imports `Networking` or `Persistence` directly. It calls `WorkoutRepository.todaysWorkout()` which returns from SwiftData immediately and refreshes from the network in the background. Makes testing trivial (swap repos for fakes) and keeps the offline path uniform.
- **Theme provider:** `ThemeStore` holds the selected coach's hue (`accentH: Int`). Cascades into every accent-using component via SwiftUI environment value. Coach switch = single state mutation.
- **Navigation:** `NavigationStack` per tab. Workout flow (in-workout + complete) uses `.fullScreenCover` since they're full-screen takeovers.
- **Concurrency:** `async/await` everywhere. Long-running work (LLM streams, Watch sessions) runs in `Task` tied to view lifecycle. No completion handlers, no Combine.

### Why this shape

- Local SPM packages give module boundaries without a workspace. Each package owns its tests.
- Repository layer is the only place that knows about online-vs-offline logic. Features stay pure UI + state.
- `@Observable` over `ObservableObject` because it's per-property granular (no over-rendering) and works cleanly with SwiftData.
- Avoiding TCA / heavy state frameworks — overkill for a personal-app v1, fights SwiftUI's natural patterns.

## 5. Watch App + HealthKit Integration

### Watch app role

Standalone watchOS SwiftUI app. It is the device the user actually interacts with mid-workout (watch on wrist; phone in bag). Phone shows the rich data-dense screen for visibility / setup; watch handles the live session.

### Watch app screens (three only)

1. **Idle:** Title + duration of today's workout, "Start" button
2. **Active Set:** Exercise name, set N/M, target reps + load, big HR readout, "Log set" button. Digital crown scrolls RPE 1–10.
3. **Rest:** Countdown ring, "Skip" button, next move preview line

### HealthKit integration

**Read side (iPhone, before workouts):**
- Types: `activeEnergyBurned`, `appleExerciseTime`, `restingHeartRate`, `heartRateVariabilitySDNN`, `sleepAnalysis`
- Computed as 7-day rolling summaries (avg HR, total active min, avg HRV, avg sleep hours), sent up with plan-generation requests
- Summaries only — no raw samples leave the device. Privacy-respecting and keeps payloads small.

**Write side (Watch, during workouts):**
- `HKWorkoutSession` started on Watch with appropriate `activityType`:
  - Strength workouts → `.traditionalStrengthTraining`
  - HIIT → `.functionalStrengthTraining` or `.highIntensityIntervalTraining`
  - Mobility/recovery → `.flexibility` or `.coreTraining`
- `HKLiveWorkoutBuilder` collects HR + active energy automatically
- On session end, builder writes one `HKWorkout` to HealthKit with metadata (duration, total energy, sets logged count)
- `WKBackgroundModes`: `workout-processing` enabled so session keeps running when watch face dims

### Phone ↔ Watch communication (`WatchConnectivity`)

| Direction | Payload | Mechanism | Rationale |
|---|---|---|---|
| Phone → Watch | Today's workout payload | `transferUserInfo` (queued, reliable) | Sent when user taps "Start" on phone, or pre-loaded earlier in the day |
| Watch → Phone | HR samples (~1 Hz) | `sendMessage` (live, reachable-only) | Powers live HR card; coalesce to ~1 Hz, drop if phone unreachable (lossy real-time is fine) |
| Watch → Phone | Logged set `{exId, setNum, reps, load, rpe}` | `transferUserInfo` (queued) | Reliable: never lose a logged set, even if phone is asleep |
| Watch → Phone | Session lifecycle (started, paused, ended) | `sendMessage` w/ `transferUserInfo` fallback | Phone UI mirrors session state |

### Edge cases designed for

- Phone unreachable mid-workout → Watch keeps logging locally, syncs when reachable. No data loss.
- Workout started on phone, watch missing → Phone falls back to manual HR entry. (BT HRM support deferred to v1.1.)
- Workout started on Watch only → Phone catches up via queued `userInfo` when it next foregrounds.
- `HKWorkoutSession` survives app backgrounding on Watch but not termination — relaunch handler detects in-progress session and resumes.

### Permissions requested at first launch

- HealthKit read (types listed above) + write (`HKWorkoutType`, `activeEnergyBurned`, `heartRate`)
- Notifications (for "tomorrow's session is ready" pings)
- (No microphone, no location, no contacts in v1)

## 6. Backend (Cloudflare Worker Proxy)

### Single file, ~80 lines

```typescript
// worker.ts (sketch)
export default {
  async fetch(req: Request, env: Env): Promise<Response> {
    if (req.method !== 'POST') return new Response('method not allowed', { status: 405 });

    if (req.headers.get('x-device-token') !== env.DEVICE_TOKEN) {
      return new Response('forbidden', { status: 403 });
    }

    const body = await req.json();

    const upstream = await fetch('https://api.anthropic.com/v1/messages', {
      method: 'POST',
      headers: {
        'x-api-key': env.ANTHROPIC_API_KEY,
        'anthropic-version': '2023-06-01',
        'content-type': 'application/json',
      },
      body: JSON.stringify({ ...body, stream: true }),
    });

    if (!upstream.ok || !upstream.body) {
      return new Response(`upstream error ${upstream.status}`, { status: 502 });
    }

    return new Response(upstream.body, {
      headers: {
        'content-type': 'text/event-stream',
        'cache-control': 'no-cache',
      },
    });
  },
};

interface Env {
  ANTHROPIC_API_KEY: string;  // Worker secret
  DEVICE_TOKEN: string;        // Worker secret
}
```

### Why this shape

- **Stateless.** No DB, no KV, no Durable Objects. Restart-safe by definition.
- **Holds the Anthropic key.** That's the only reason it exists. Without this, the iOS binary would have to ship the key, which is extractable.
- **Shared-secret auth.** `X-Device-Token` is a long random string baked into both the iOS build and the Worker secret. Single user, single device-class — this is sufficient. (When opening up to multi-user, swap to per-user JWT.)
- **Workers Analytics gives free observability** — request count, error rate, p99 latency, geographic distribution. No additional setup.

### Configuration

- Worker name: `pulse-proxy`
- Env vars (set via `wrangler secret put`): `ANTHROPIC_API_KEY`, `DEVICE_TOKEN`
- Custom domain (optional): `proxy.pulse.local` or use the default `pulse-proxy.<subdomain>.workers.dev`

### What the Worker does NOT do (intentionally)

- No request logging that captures user content (privacy)
- No rate limiting (single user — limited by Anthropic's per-key quotas)
- No prompt assembly (iOS owns this)
- No response transformation (passes Anthropic's stream through verbatim)
- No retries (iOS handles retry-with-backoff)

## 7. AI Integration

### Three call sites, three model choices

| Call site | Model | Streaming | Cache strategy | Why |
|---|---|---|---|---|
| Plan generation | Opus 4.7 | SSE | Cache system prompt | Heavy reasoning, infrequent (~3–7×/week), quality matters most |
| Plan adaptation | Opus 4.7 | SSE | Cache system prompt + prior plan | Same kind of reasoning, runs after every workout |
| Coach chat (v1.1) | Sonnet 4.6 | SSE | Cache system prompt + full history | High frequency, latency-sensitive, deep cache savings |

### Plan generation prompt structure

```
[system, cached]
You are {coach.name}, {coach.style}.
You design adaptive workout programs based on the user's profile, recent
training history, and physiological signals. You output a single JSON object
matching the schema below. Stream checkpoint markers as ⟦CHECKPOINT: <label>⟧
during reasoning so the UI can show progress. Final output must be a valid
JSON object inside a ```json code block.

[Programming principles: periodization, equipment substitution rules, RPE/load
progression, recovery heuristics, injury contraindications.]

[Available exercises: <catalog manifest IDs + names + equipment>]

[Output schema for v1: Workout { id, title, subtitle, type, durationMin,
intensity, blocks, exercises, why, whispers }]
[Schema evolves in v1.1 to: WorkoutPlan { weekStart, days: [Workout{...}] }]

[user, not cached]
Profile: {goals, level, equipment, frequency, injuries}
Recent 7d HealthKit summary: {avgHRV, sleepHrs, restingHR, totalActiveMin}
Last 5 sessions: [{title, intensity, rating, mood, tags, avgHR, completed}]
Today: {date, dayOfWeek}
[v1.1: Current plan if exists: {plan_payload}]

Generate today's workout. Stream checkpoints as you reason.
```

In v1.1, the prompt expands to "Generate this week's plan" with the weekly schema; the iOS code paths for streaming + parsing are identical.

### Adaptation prompt structure

```
[system, cached — same coach prompt as planning]

[user]
Most recent session: {workout payload + set logs + WorkoutFeedback}
Current week plan: {plan_payload}

Update the next 1–3 scheduled sessions based on the user's feedback. Output:
1. Brief rationale (1–2 sentences in {coach.name}'s voice)
2. ⟦ADJUSTMENT⟧ markers for each change: {label, detail, affectedDay}
3. Updated plan JSON

Use these as guidance, not rules:
- intensity ≥ 4 → reduce load
- intensity ≤ 2 → increase load or add work
- mood=rough or low_energy tag → swap to recovery
- form_struggled → tempo work, lower load
- down-rated exercises → substitute with equivalent moves
- fresh_moves tag → rotate in 2–3 new exercises
```

The README's adjustment heuristics become *guidance to Claude*, not client-side derivations. The model returns the actual diff, iOS renders it. This is the "real adjustments, not pre-canned strings" call-out in the design handoff.

### Streaming protocol (SSE)

The Worker passes Anthropic's native SSE through. The iOS `SSEStreamParser` consumes the stream and surfaces three event types to the UI layer:

```
event: checkpoint   → { label: string }                    (rendered as mono console line)
event: adjustment   → { label, detail, affectedDay }       (adaptation only; appended to list)
event: done         → { plan: PlanJSON, tokens: {in, out} } (terminal)
```

iOS extracts `⟦CHECKPOINT: ...⟧` and `⟦ADJUSTMENT: ...⟧` markers from the model's text content as they stream. The final ` ```json ... ``` ` code block is parsed and validated against a Codable struct on `done`.

### Mid-workout coach whispers

Per the chosen architecture, whispers are pre-generated when the plan is generated. The plan-generation Opus call also produces 2–4 whisper lines per exercise (cued by set number or HR zone) which ship as part of the workout payload. iOS picks one to display at the right moment based on local triggers (set 1 start, after high-RPE set, mid-rest). No additional LLM call at workout time → zero gym-Wi-Fi dependency for cues.

### Hot-loaded system prompts

System prompts live as a JSON file in R2 (`prompts.json`):
```json
{
  "version": 7,
  "coachPrompts": {
    "ace": "...",
    "rex": "...",
    "vera": "...",
    "mira": "..."
  },
  "planGenerationFraming": "...",
  "adaptationFraming": "..."
}
```

iOS fetches at app launch with stale-while-revalidate (cached in SwiftData, refreshed in background, used immediately even if stale). Lets you tweak prompts without shipping an app update.

### Output validation

Every plan/adaptation response is decoded into a Codable schema (`WorkoutPlan`, `PlanAdaptation`). On decode failure:
1. One silent retry with a stricter "respond with valid JSON only" reminder appended to the prompt
2. If the retry still fails, surface "Generation failed, using fallback workout"
3. Fallback: a hand-authored "default day" baked into the app bundle (squats, push-ups, plank, mobility) so the user can always train

### Cost ballpark (single user, monthly)

- Plan generation: ~16 calls/mo × ~$0.20 = $3.20
- Adaptation: ~16 calls/mo × ~$0.10 = $1.60
- Chat (v1.1): ~$0.50
- **Total: ~$5–7/mo at current Anthropic pricing**

### Safety guardrails

- System prompts include scope limits: "Never recommend exercise volumes above [tier] for users marked `level=new`", "If the user reports injury, route to recovery and surface a 'see a professional' note"
- No medical claims, no nutrition or supplement advice — explicitly out of scope in system prompt
- The fallback workout is non-injurious: bodyweight, no load progression, common movements only

## 8. Content Pipeline (Exercise Demos)

### Source: Free Exercise DB

[Free Exercise DB](https://github.com/yuhonas/free-exercise-db) — public-domain (Unlicense) exercise library, 800+ exercises. Each entry has:
- Name, primary/secondary muscles, equipment, level, force, mechanic, category
- Two photo demonstrations (start position + end position) as JPEGs in the repo
- Step-by-step instructions array

License is Unlicense (public domain) — no attribution required, commercial use allowed, closed-source distribution allowed. This keeps the door open to commercializing or opening up to multi-user later without licensing headaches.

### One-time import script (run locally on Mac)

Not part of the iOS app or Worker — a standalone Node/TypeScript script (`tools/import-exercises/`) you run once to populate R2.

```
1. Fetch free-exercise-db dist/exercises.json (the curated combined file)
2. For each exercise:
   a. Download images (typically 2 photos per exercise: start + end position)
      from raw.githubusercontent.com/yuhonas/free-exercise-db/main/exercises/{id}/
   b. Use ffmpeg to combine the 2 frames into a 2-second looping MP4
      (1.5s per frame, slight crossfade, H.264, 720×720, muted)
   c. Save the first image as a poster JPEG
   d. Upload MP4 → r2://pulse/exercises/{id}.mp4
   e. Upload poster → r2://pulse/exercises/{id}-poster.jpg
3. Build manifest JSON keyed by exercise id, with all metadata + URLs
4. Upload manifest → r2://pulse/exercises/manifest.json
```

Total time: ~30–60 minutes for 800 exercises (script-driven, runs unattended). Storage: ~1–2 GB in R2 (~$0.02/mo).

### Why MP4 over the original JPEGs

- AVPlayer renders MP4 natively at 60 fps; matches the design language of "demonstrations in motion"
- 2-frame oscillation between start/end position is visually informative — better than a static image, simpler than animated 3D models
- Easier to loop seamlessly via `AVPlayerLooper`
- Same playback code path as if we later upgrade specific exercises to AI-generated cinematic clips

### iOS playback

- `AVPlayerLayer` inside the exercise card. Autoplay + muted + looped via `AVPlayerLooper`.
- **Pre-fetch:** when a workout is loaded into the In-Workout screen, kick off background `URLSession` downloads for every exercise's video into the on-device cache (`Library/Caches/exercises/`). Mid-workout playback hits cache, never network.
- **Cache management:** cap at 250 MB; LRU eviction. Common exercises stay cached forever in practice.
- **Failure mode:** if a video fails to load (no network, R2 outage, missing asset), fall back to the `<EP>` Exercise Placeholder from `DesignSystem` — already designed for this.

### Catalog constraint

Claude's plan-generation system prompt is constrained to exercises in the current manifest:
> "You may only select exercises from this catalog. Each entry has an ID, name, equipment requirements, and target body part. Output exercises by their ID."

This guarantees every exercise in a generated plan has a matching video asset on day one. When you add new exercises (re-running the import or hand-curating later), the manifest version bumps and iOS fetches the update.

### Future enhancement (post-v1)

For your top 20–30 most-used exercises, you can generate cinematic Pulse-branded clips using Sora/Veo with a curation gate, then swap them in by updating individual `videoURL` entries in the manifest. No app changes required; the schema supports per-exercise URL changes.

## 9. Data Model (SwiftData)

All on-device. Source of truth in v1. Every model carries an optional `userID: UUID?` for future multi-user readiness.

```swift
@Model final class Profile {
    var id: UUID
    var userID: UUID?              // multi-user readiness, nil in v1
    var displayName: String
    var goals: [String]
    var level: String              // "new" | "regular" | "experienced" | "athlete"
    var equipment: [String]
    var frequency: Int             // sessions/week
    var weeklyTargetMinutes: Int
    var activeCoachID: String      // "ace" | "rex" | "vera" | "mira"
    var accentHue: Int             // denormalized from coach for fast theme lookup
    var createdAt: Date
}

@Model final class Plan {
    var id: UUID
    var userID: UUID?
    var weekStart: Date
    var generatedAt: Date
    var modelUsed: String
    var promptTokens: Int
    var completionTokens: Int
    @Attribute(.externalStorage) var payloadJSON: Data  // full WorkoutPlan blob
}

@Model final class Workout {
    var id: UUID
    var userID: UUID?
    var planID: UUID
    var scheduledFor: Date
    var title: String
    var subtitle: String
    var workoutType: String        // "Strength" | "HIIT" | "Mobility" | etc.
    var durationMin: Int
    var status: String             // "scheduled" | "in_progress" | "completed" | "skipped"
    @Attribute(.externalStorage) var blocksJSON: Data
    @Attribute(.externalStorage) var exercisesJSON: Data
    @Attribute(.externalStorage) var whispersJSON: Data?  // pre-generated coach cues
}

@Model final class Session {
    var id: UUID
    var userID: UUID?
    var workoutID: UUID
    var startedAt: Date
    var completedAt: Date?
    var avgHR: Int?
    var kcal: Int?
    var durationSec: Int?
    var watchSessionUUID: UUID?    // mirrors HKWorkoutSession identifier
    @Relationship(deleteRule: .cascade) var setLogs: [SetLog]
    @Relationship(deleteRule: .cascade, inverse: \Feedback.session) var feedback: Feedback?
}

@Model final class SetLog {
    var sessionID: UUID
    var exerciseID: String
    var setNum: Int
    var reps: Int
    var load: String               // string to allow "20 kg", "BW", "0:30"
    var rpe: Int                   // 1–10
    var loggedAt: Date
    var session: Session?

    // Composite uniqueness enforced in repository writes:
    // (sessionID, exerciseID, setNum) — idempotent
}

@Model final class Feedback {
    var id: UUID
    var userID: UUID?
    var session: Session?
    var submittedAt: Date
    var rating: Int                // 1–5
    var intensity: Int             // 1–5
    var mood: String               // "great" | "good" | "ok" | "rough"
    var tags: [String]
    @Attribute(.externalStorage) var exRatingsJSON: Data  // [exId: "up"|"down"|null]
    var note: String?
}

@Model final class Adaptation {
    var id: UUID
    var userID: UUID?
    var feedbackID: UUID
    var appliedToPlanID: UUID
    var generatedAt: Date
    var modelUsed: String
    var promptTokens: Int
    var completionTokens: Int
    @Attribute(.externalStorage) var diffJSON: Data
    var rationale: String
}

@Model final class ChatMessage {           // v1.1
    var id: UUID
    var userID: UUID?
    var coachID: String
    var role: String                       // "user" | "assistant"
    var content: String
    var sentAt: Date
}

@Model final class ExerciseAsset {
    var id: String                         // matches catalog manifest
    var name: String
    var focus: String
    var level: String
    var kind: String
    var equipment: [String]
    var videoURL: URL
    var posterURL: URL
    var instructionsJSON: Data
    var manifestVersion: Int               // for cache invalidation
}
```

### Schema rationale

- `payloadJSON` / `blocksJSON` / `exercisesJSON` stored as `Data` (with `.externalStorage`) because LLM-generated plan shape will evolve as prompts change. Hot fields (status, scheduledFor, workoutType) extracted as columns for indexing.
- Cascade delete on `Session → SetLog/Feedback`: if you delete a session, its records go too.
- `(sessionID, exerciseID, setNum)` composite uniqueness enforced in `SessionRepository.logSet(...)` (SwiftData lacks compound unique constraints in v1, so the repository handles it via fetch-then-update or insert).

### Migration strategy

- v1 → v1.1: add `ChatMessage`, no breaking changes
- v1 → multi-user: backfill `userID` on all rows from the single Profile, add a `User` entity, possibly migrate to CloudKit-backed SwiftData for sync. Schema doesn't change.

## 10. Cross-Cutting Concerns

### Error handling — three tiers

**Tier 1 — Network / LLM outages (recoverable, common):**
- Plan generation fails → keep last plan visible, banner: "Couldn't refresh — using last plan"
- Adaptation fails → toast, queue retry on next app open, plan unchanged
- All API calls wrapped in retry-with-backoff (3 attempts, 1s/3s/9s)
- **Mid-workout: everything works offline.** HR streaming, set logging, video playback (cached), timers — all local. No network calls during a session.

**Tier 2 — Malformed LLM output (rare, must catch):**
- Every plan/adaptation validated against Codable schema
- Invalid JSON → one silent retry with stricter prompt → if still bad, fall back to hand-authored default workout
- Default workout: bundled in app, always available, conservative

**Tier 3 — Crashes / data corruption (must not lose state):**
- `SessionStore` auto-persists to SwiftData on every state change
- App relaunch mid-workout → restore at exact set, with timer offset by elapsed wall-clock
- `HKWorkoutSession` independently persistent — survives Watch app crash + relaunch
- Set log writes idempotent on `(sessionID, exerciseID, setNum)` — replay-safe

### Testing strategy

| Layer | Test type | Why |
|---|---|---|
| Repositories | Unit, with fakes for Networking + HealthKit | Critical: all features depend on them |
| `SSEStreamParser` + checkpoint extractor | Unit | Easy to break, hard to debug live |
| Plan/adaptation JSON schemas | Unit, decode real recorded responses | Catches Codable drift |
| In-Workout screen across phases | SwiftUI snapshot tests | Catches accidental layout regressions |
| End-to-end happy path | One scripted UI test (onboarding → workout → complete → adaptation) | Run before every TestFlight build |
| Calendar / Library / Stats | None (deferred to v1.1) | n/a |
| Real Anthropic API in CI | None — mock at SSE-stream level | Cost + flakiness |
| Watch app | Smoke test only; manual on-device | watchOS testing tooling is poor |

### Observability

- **Sentry** (free tier, 5k events/mo) for iOS crashes. `sentry-cli` symbol upload in build phase.
- **Cloudflare Workers Analytics** for proxy traffic — request count, errors, latency, no setup needed.
- **In-app debug panel** reachable by 5-tap on profile avatar. Shows: last 20 LLM calls (model, tokens, ms), SwiftData stats, HealthKit auth status, Watch reachability, R2 manifest version. Saves debugging blind.
- **Skip:** product analytics (Mixpanel/PostHog), A/B testing, attribution. Single user — none of it applies.

### Privacy

- HealthKit raw samples never leave the device. Only aggregated 7-day summaries (avg HR, total active min, HRV mean) sent to Claude.
- No analytics SDKs, no ad SDKs, no third-party trackers. Sentry only for crashes (no PII).
- Coach chat history (v1.1) stays on-device.
- `Info.plist` declares HealthKit + (future) microphone usage strings honestly.

### Deployment

- **iOS:** Xcode → Archive → upload to App Store Connect → TestFlight internal track. No App Store review (internal only).
- **Worker:** `wrangler deploy` from worker repo. Single command, ~5 second deploy.
- **R2 manifest / videos:** uploaded once via import script; updated rarely. No CI.
- **Versioning:** semver on iOS app. No versioning on Worker (idempotent stateless function).
- **No CI/CD pipeline in v1.** Solo dev shipping to themselves. Add GitHub Actions for the Worker if you want auto-deploy on merge.

### Distribution

- Apple Developer Program ($99/year)
- TestFlight internal testing track only — installable on your devices indefinitely, no review process
- Keep the option open to invite a friend later via TestFlight external testing (requires a one-time review of that build)
